# TikTok ROI 智能运营平台 — MVP 产品设计

> 面向读者：孙栩尧（主用户）、赵总（决策者）、李远哲（商务）、张翼翔（开发）
> 版本：v1.0（2026-04-07，基于所有既有 md 与团队对齐结果）

---

## 一、一句话定位

> "运营在选品/上品阶段，输入货源成本和产品信息，系统就输出**这个品该定多少钱、能不能做、ROI 多少、竞品卖多少**，并把决策结果存档复用。"

## 二、核心判断（为什么 MVP 长这样）

基于《会议纪要-TikTok电商运营流程与AI改造方案.md》确立的两个方向性结论：

1. **整套运营逻辑围绕 ROI 运转**，ROI ≥ 2.5 是保本硬指标。
2. **AI 的最大价值不在操作自动化，而在决策支持**——帮运营算清楚"该怎么定价、该选哪个达人、该投多少流"。

因此 MVP **不是**"把所有痛点都沾一点"，而是**聚焦"定价决策"这一个最痛的点**，做深做透，把"Excel 算不清 + 凭经验猜 + 没历史可查"三件事一次性解决。

---

## 三、产品形态

### 形态：**单一 Web 应用 "ROI 定价决策工作台"**

| 方案 | 取舍 |
|---|---|
| ✅ **Web（采用）** | 研发成本最低、迭代最快、无需审核；运营人员只需浏览器；与 FastMoss/TikTok API 直连；多端访问 |
| ❌ 浏览器插件 | 必须依附 TikTok 后台 DOM，接口一变就坏；"自动填字段"价值低，延后到 Phase 4 |
| ❌ 原生 App | 主用户坐电脑前工作，移动场景不是 MVP 痛点 |
| ❌ Excel 模板 | 算得清但联不上 API、存不住历史、无法沉淀数据资产 |

### 用户与角色（MVP 只做 3 种）

| 角色 | 能做什么 |
|---|---|
| 运营（孙栩尧，主用户） | 新建/编辑商品、算 ROI、看竞品、保存定价方案、复盘历史决策 |
| 老板（赵总，只读） | 看 Dashboard：当日 GMV / 总 ROI / 盈亏 / 已存档商品列表 |
| 商务/管理员（李远哲） | 管理用户、切换店铺、查看 API 调用额度 |

### 主界面结构（5 个页面）

```
┌─ 顶部导航：店铺选择 / 用户菜单
│
├─ 1. Dashboard（看板）
│    · 本日 GMV / 订单数 / 店铺总 ROI 实测值
│    · FastMoss 调用余量 / 本月成本
│    · 最近 10 条定价决策记录
│
├─ 2. 商品工作台（核心）⭐ MVP 主阵地
│    · 左：商品列表（我保存过的所有"候选/在售"商品）
│    · 右：详情页 = 成本四件套 + ROI 计算器 + 竞品分析 + 定价建议
│
├─ 3. 竞品探索
│    · 按关键词/类目查 FastMoss topSelling / product search
│    · 价格带分布图、销量分布、点进去看单品详情
│    · 一键"作为参考加入某商品的竞品池"
│
├─ 4. 运费与规则
│    · 运费表维护（普/特/敏货费率，支持后台更新）
│    · ROI 保本线、默认佣金率等全局参数配置
│
└─ 5. 设置
     · 店铺 OAuth 绑定、API key 管理、成员邀请
```

## 四、MVP 必须包含的 7 个功能（F1–F7）

### F1. 运费计算器 · 2 天
- 输入：产品重量 / 货物类型（普/特/敏）/ 目的国（MVP 固定日本）/ 是否包邮
- 输出：单件物流成本、不同打包组合（1 件/5 件/10 件装）的运费对比
- 运费表在"运费与规则"页后台维护，避免代码改动
- **复用点**：会议纪要公式 `重量×每公斤单价 + 基础包裹运费（475 日元）`

### F2. ROI 计算器 & 盈亏模拟 · 3 天
- 输入四项成本：货源成本 / 物流（来自 F1）/ 投流预算 / 达人佣金率
- **双向计算**：
  - 正向：给定售价 → 算 ROI
  - 反向：给定目标 ROI（默认 2.5）→ 反推最低售价
- 输出：零售价 / 折扣价 / 秒杀价 三档建议，盈亏平衡点
- 多 SKU 支持：单装 / 5 装 / 10 装分别算一组

### F3. 竞品价格带分析 · 3 天
- 调用 FastMoss：
  - `/product/v1/search`（关键词搜索）
  - `/product/v1/rank/topSelling`（热销榜，按日本市场）
  - `/product/v1/goodSalesList`（历史销量）
- 展示：价格分布直方图、Top 20 对标品、销量/价格散点图
- "加入参考池" → 在 F4 生成建议时作为上下文

### F4. LLM 智能定价建议 · 3 天
- 基于 F2 的成本结构 + F3 的竞品数据 + 产品描述，调用 Claude Sonnet 4.6
- 输出**结构化建议**：
  - 建议零售价 / 折扣价 / 秒杀价（带理由）
  - 市场定位（低价引流 / 中位跟随 / 高价差异化）
  - 风险提示（如：竞品均价远低于保本价 → 建议弃坑）
- 同一产品可多次生成，保留历史建议对比
- 通过阿里云 Proxy 转发 Claude API

### F5. 商品工作台（CRUD + 决策存档）· 3 天（UI 共用）
- 创建 / 编辑 / 归档商品卡
- 每个商品保存：基础信息 · 成本结构 · 竞品池 · 历史定价建议 · 最终上架价
- **这是数据资产沉淀的核心**：同一个品下次调整就不用重来
- 是 MVP 比 Excel 唯一的核心差异点，**必须死守**

### F6. 店铺自测 Dashboard · 2 天
- 通过 TikTok Shop Partner API v2 拉自店数据：当日订单、GMV、店铺实测 ROI
- 与 F4 建议值做对比，回填"建议 vs 实际"偏差
- **闭环价值**：让用户看到"AI 建议的定价在真实店铺跑出来 ROI 是多少"
- 兜底方案：若 OAuth 审批延迟，先支持订单 CSV 手动导入

### F7. 1688 图搜反查货源 · 1 天
- 万邦 Onebound `item_search_img`：上传一张 TikTok 竞品图 → 返回 1688 同款货源价
- 作为 F2 货源成本的"外部参考值"
- Pangolin 备源在 MVP 可以只搭抽象层、不接实现

## 五、MVP 明确不做的功能

| 功能 | 原因 | 归属 |
|---|---|---|
| AI 达人评估系统 | 达人接口 $0.02/次 贵 2 倍；模型需数据积累期 | Phase 2 |
| 投流优化助手 | 依赖 Ads API 审批 + 至少 1 个月投流历史数据 | Phase 3 |
| 标题/描述/视频 AI 生成 | 豆包等工具暂能凑合，非最痛点 | Phase 2 |
| 浏览器插件（自动填字段） | 依附 DOM 脆弱，决策价值 >> 录入价值 | Phase 4 |
| 订单智能分拣 | 订单量没到瓶颈（日出百单才痛），不影响决策 | Phase 4 |
| 活动到期提醒 / 续期 | 靠人肉日历暂能扛 | Phase 4 |
| 官方活动推荐 | 规则复杂、数据闭环弱 | Phase 4 |
| 多店铺 / 多租户 | MVP 就一家店，数据模型预留字段即可 | Phase 2+ |

## 六、关键设计决策

1. **"决策存档"是灵魂**：每次定价建议都要可追溯、可复盘。这是 MVP 比 Excel 强的唯一理由，必须死守。
2. **日本市场单区域**：MVP 不做多国参数化，硬编码日元、日本运费表即可。
3. **FastMoss 免费 500 次 + 缓存层**：竞品查询结果按"关键词+日期"缓存 24h，避免重复计费；MVP 联调期在免费额度内即可跑完。
4. **LLM 只用 Sonnet 4.6**：不引入 Haiku / 本地模型，减少复杂度；开启 Prompt Caching 控成本。
5. **Dashboard 不做大屏**：MVP 只有 3 张卡（GMV / ROI / 余量）+ 1 张决策列表，不做花哨图表。

## 八、产品验收方式（非技术验收）

把本方案拿给：

1. **孙栩尧（主用户）** — F1–F6 是否覆盖"新品上架前最痛的那 2 小时"？
2. **赵总** — Dashboard 的 3 张卡是他关心的指标吗？需要加"店铺月净利润"吗？
3. **李远哲** — 25 天工期 + ¥5,550–7,730 非人力成本 + 砍掉 4 个模块，是否能形成可卖的 MVP 报价？

### MVP 量化验收指标（待孙栩尧/赵总补充）
- 定价决策耗时：**从目前 ~120 分钟/品 降到 ≤ 20 分钟/品**
- ROI 预测偏差率：**AI 建议价 vs 实跑 ROI，偏差 ≤ 15%**
- 决策存档命中率：**≥ 50% 的新品决策能复用已存档商品的参数**

## 九、风险

| 风险 | 影响 | 缓解 |
|---|---|---|
| 运营看到 MVP 觉得"就这？没达人没投流没自动化" | 用户期望管理 | 本方案提前 review，明确 Phase 2 何时上线 |
| FastMoss 日本市场数据深度不够 | F3 竞品分析失去意义 | Phase 0 用 500 次免费额度先验证数据质量 |
| TikTok Partner API OAuth 审批延迟 | F6 无法按时上线 | 先做 F1–F5，F6 作为第 5 周交付；兜底 CSV 导入 |
| LLM 建议不准，用户不信 | F4 沦为摆设 | 建议页必须展示**推理过程**（成本结构 + 竞品摘要），不只是结果 |
| 需求蔓延（孙栩尧/赵总临时加需求） | 工期超支、MVP 范围失控 | 严格按本文"不做"清单执行，新需求走 Phase 2 |
