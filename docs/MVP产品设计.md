# TikTok ROI 智能运营平台 — 一阶段产品设计

> 面向读者：孙栩尧（主用户）、创世跨境（决策者）、李远哲（商务）、张翼翔（开发）
> 版本：v2.0（2026-04-10，基于合同 Phase 1 验收标准 + 接口实测 + 客户确认）

---

## 一、一句话定位

> "运营粘贴一条 TikTok 商品链接，系统 3 分钟内输出一份完整的定价决策报告——该卖多少钱、成本多少、竞品卖多少、找哪些达人、ROI 能不能过 2.5——并把每一次决策存档可复盘。"

## 二、核心判断（为什么产品长这样）

1. **整套运营逻辑围绕 ROI 运转**，ROI ≥ 2.5 是保本硬指标。
2. **AI 的最大价值不在操作自动化，而在决策支持**——帮运营算清楚"该怎么定价"。
3. **不是工具箱，是一条管线**——运营不想在 7 个页面之间跳来跳去，他只想粘贴一个链接，拿到一份报告。

因此产品形态是**单页报告生成器**，不是多页面工作台。

---

## 三、产品形态

### 形态：单一 Web 应用 — 单页报告式工作流

| 方案 | 取舍 |
|---|---|
| ✅ **单页报告生成器（采用）** | 粘贴链接 → 系统自动跑完 F1-F7 → 输出一页可滚动的决策报告；运营只需做两件事：粘贴 + 审阅 |
| ❌ 多页面工作台 | 需要在多个页面跳转拼凑信息，心智负担重 |
| ❌ 浏览器插件 | 依附 TikTok 后台 DOM，接口一变就坏；延后到 Phase 4 |
| ❌ 原生 App | 主用户坐电脑前工作，移动场景不是一阶段痛点 |

### 用户与角色（一阶段 3 种）

| 角色 | 能做什么 |
|---|---|
| 运营（孙栩尧，主用户） | 粘贴链接 → 生成报告 → 补充信息 → 重新生成 → 存档 → 回填实测数据 |
| 老板（创世跨境，只读） | 看存档的报告列表 + 报告详情；看"AI 建议 vs 实跑 ROI"偏差 |
| 管理员（李远哲） | 管理用户、全局参数配置（ROI 保本线、退货率、汇率）、API 额度查看 |

### 核心工作流（一句话版）

```
粘贴 TikTok 链接 → 系统自动生成报告 → 运营审阅/补充 → 存档 → 上架后回填实测 → 复盘
```

### 信息层级（为什么这个顺序，不是 F1→F7）

| 顺序 | Section | 原因 |
|---:|---|---|
| 1 | F4 AI 定价建议 | 运营的核心问题是"该卖多少钱"，**结论先给** |
| 2 | F2 ROI 计算 | 量化支撑 — "为什么是这个价" |
| 3 | F6 1688 货源 | 成本侧 — "进价到底多少" |
| 4 | F1 运费 | 成本侧 — "运费多少" |
| 5 | F3 竞品 | 市场侧 — "别人卖多少" |
| 6 | F7 达人 | 补充 — "投达人还要花多少" |
| 7 | 补充信息 | 修正 — "系统算错的地方我来改" |
| 8 | Phase 2/3 预览 | 未来展望 |

## 六、外链直达原则

所有引用的外部数据**必须带可点击的原始来源链接**，运营可以一键跳转到数据源验证。

| 数据类型 | 外链格式 | 来源字段 |
|---|---|---|
| **1688 货源候选** | `https://detail.1688.com/offer/{num_iid}.html` | Onebound `item_get` 返回的 `detail_url` |
| **TikTok 竞品商品** | 原始 TikTok 商品页 | FastMoss `/product/v1/search` 返回的 `tiktok_url` |
| **FastMoss 竞品分析** | FastMoss 商品分析页 | FastMoss 返回的 `fastmoss_url` |
| **TikTok 达人主页** | `https://www.tiktok.com/@{unique_id}` | FastMoss `/creator/v1/search` 返回的 `unique_id` |

设计要求：所有外链统一用 🔗 图标标识，点击在新 tab 打开。

## 七、数据流（并行 vs 串行）

```
用户粘贴 TikTok 链接
       ↓
解析 product_id + 主图 URL + 标题
       ↓ 并行触发（F6/F3/F7 互不依赖）
       ├── F6: 主图 → Onebound item_search_img → Top 10 候选 → item_get 取详情
       ├── F3: 标题提取关键词 → FastMoss search + topSelling + goodSalesList
       └── F7: 竞品 product_id → FastMoss creatorList + creator search/productList
       ↓ F6 完成后
       F1: 重量(F6) + 货品类型(手选) → 运费计算
       ↓ F1 + F6 + F7 完成后
       F2: 货源成本(F6) + 运费(F1) + 佣金(F7) + 投流(手填) → ROI 计算 + 红线
       ↓ F2 + F3 + F7 完成后
       F4: Claude Sonnet 4.6(F2成本 + F3竞品 + F7达人 + 产品描述) → AI 定价建议
```

**加载体验**：
- 提交链接后，各 section 以 skeleton loading 按顺序出现
- F6/F3/F7 并行加载，哪个先完成哪个先显示
- F4（AI 建议）最后加载，因为依赖所有其他模块的数据
- 进度提示：`正在获取竞品数据...` / `正在查询 1688 货源...` / `正在生成 AI 建议...`

## 八、"补充信息 → 重新生成"机制

这是本产品的核心差异化：**系统不是一锤子买卖，而是允许运营用自己的 know-how 修正系统的输出**。

### 流程

1. 系统用 API 数据自动生成 v1 报告
2. 运营发现"进价不对 / 重量不对 / 佣金率不是这个"
3. 在「补充信息」区覆写对应字段
4. 点击 [重新生成]
5. 系统保存 v1 → 用覆写值重跑受影响模块（不重跑未变化的）→ 生成 v2
6. 顶部版本选择器：可在 v1 / v2 之间切换
7. 变化字段黄色高亮，方便对比

### 版本管理

- 每次重新生成创建新版本（v1, v2, v3...）
- 所有版本共享同一 product_id
- 版本选择器在报告顶部，下拉切换
- 支持两个版本并排对比（split view，变化字段黄色高亮）

## 九、F5 决策存档 — 持久化层（不是报告里的一个 section）

F5 是所有报告的**存储后端**。每份报告就是一条 F5 记录。

### 存档内容

点击 [存档] 时，系统创建一条商品卡：
- 基础信息（TikTok 链接、product_id、标题、主图）
- F6 货源快照（所有候选 + 选中的那个）
- F1 运费快照
- F3 竞品池快照（Top 20 + 图表数据）
- F7 达人预估快照（Top 20 达人 + 佣金计算）
- F2 ROI 计算全量输入/输出
- F4 AI 建议（全版本）
- 最终上架价（存档时填或后续回填）
- 所有快照**必须在查询当时入库**，事后补齐不在一阶段范围

### 运营回填实测数据

商品上架后，运营可以在存档报告上回填：
- 最终上架价
- 上架后 N 天实测 GMV
- 实跑 ROI
- 实际广告花费
- 实际达人分佣

系统自动算出「AI 建议 vs 实际」偏差。这是 Phase 2 达人评估/投流优化的训练数据基础。

### 报告列表页

一个轻量级页面（侧边抽屉或独立页），显示所有存档报告：
- 商品缩略图 + 标题
- 存档日期
- AI 建议价 vs 实际上架价（如已回填）
- 状态：`待上架` / `已上架` / `已下架`
- 快捷操作：打开报告 / 回填数据

## 十一、关键设计决策

1. **结论先行**：F4 AI 建议放在报告最顶部，运营一眼看到"该卖多少"，再往下看为什么
2. **一页到底**：不做多页面跳转，所有信息在一个可滚动页面上，减少认知负担
3. **外链直达**：每条货源/竞品/达人都带原始链接，运营随时可以跳到 1688/TikTok/FastMoss 验证
4. **允许修正**：报告生成后不是只读的，运营可以用自己的 know-how 覆写数据并重新生成
5. **版本可比**：每次重新生成保留历史版本，支持并排对比
6. **快照入库**：所有外部数据在查询当时就写入数据库，不依赖"以后再查"
7. **AI 建议准不准 ≠ 验收标准**：合同 §6.1.2.2 明确，靠 F5 回填积累数据后 Phase 2 再验证

## 十三、验收方式

### 工作流验收（端到端）

拿 3-5 个真实 TikTok 商品链接走完整流程：
1. 粘贴链接 → 自动生成报告
2. 报告各 section 数据正确、外链可点击
3. 在「补充信息」覆写进价/重量 → 重新生成 → 版本对比正常
4. 存档 → 报告列表可见
5. 回填实测数据 → 偏差自动计算

### 量化验收指标

- 定价决策耗时：目标 ≤ 20 分钟/品（当前约 120 分钟）
- 决策存档命中率：≥ 50% 新品能复用已存档参数
- F3/F6/F7 数据流稳定性 ≥ 99%
- **不含"AI 建议准确率"指标**（合同 §6.1.2.2 明确）

## 十四、风险

| 风险 | 影响 | 缓解 |
|---|---|---|
| FastMoss 日本达人覆盖度 < 1,000 | F7 降级为"列 Top 50" | Phase 0 quota 预测试，合同已写降级条款 |

| FastMoss 关键词只支持单 token | F3 搜索精度受限 | [修改关键词] 按钮让运营手动调整 |
| LLM 建议不准 | 运营不信 F4 | 展示完整推理过程（不只给结果）+ F5 回填积累校准数据 |
| 需求蔓延 | 工期超支 | 严格按合同 §6.1.2 的 25 项不含清单执行 |
