# TikTok ROI 智能运营平台 — MVP 产品设计

> 面向读者：孙栩尧（主用户）、赵总（决策者）、李远哲（商务）、张翼翔（开发）
> 版本：v1.0（2026-04-07，基于所有既有 md 与团队对齐结果）

---

## 一、一句话定位

> "运营在选品/上品阶段，输入货源成本和产品信息，系统就输出**这个品该定多少钱、能不能做、ROI 多少、竞品卖多少**，并把决策结果存档复用。"

## 二、核心判断（为什么 MVP 长这样）

基于《会议纪要-TikTok电商运营流程与AI改造方案.md》确立的两个方向性结论：

1. **整套运营逻辑围绕 ROI 运转**，ROI ≥ 2.5 是保本硬指标。
2. **AI 的最大价值不在操作自动化，而在决策支持**——帮运营算清楚"该怎么定价、该选哪个达人、该投多少流"。

因此 MVP **不是**"把所有痛点都沾一点"，而是**聚焦"定价决策"这一个最痛的点**，做深做透，把"Excel 算不清 + 凭经验猜 + 没历史可查"三件事一次性解决。

## 三、产品形态

### 形态：**单一 Web 应用 "ROI 定价决策工作台"**

| 方案 | 取舍 |
|---|---|
| ✅ **Web（采用）** | 研发成本最低、迭代最快、无需审核；运营人员只需浏览器；与 FastMoss/TikTok API 直连；多端访问 |
| ❌ 浏览器插件 | 必须依附 TikTok 后台 DOM，接口一变就坏；"自动填字段"价值低，延后到 Phase 4 |
| ❌ 原生 App | 主用户坐电脑前工作，移动场景不是 MVP 痛点 |
| ❌ Excel 模板 | 算得清但联不上 API、存不住历史、无法沉淀数据资产 |

### 用户与角色（MVP 只做 3 种）

| 角色 | 能做什么 |
|---|---|
| 运营（孙栩尧，主用户） | 新建/编辑商品、算 ROI、看竞品、保存定价方案、复盘历史决策 |
| 老板（赵总，只读） | 看 Dashboard：当日 GMV / 总 ROI / 盈亏 / 已存档商品列表 |
| 商务/管理员（李远哲） | 管理用户、切换店铺、查看 API 调用额度 |

### 主界面结构（5 个页面）

```
┌─ 顶部导航：店铺选择 / 用户菜单
│
├─ 1. Dashboard（看板）
│    · 本日 GMV / 订单数 / 店铺总 ROI 实测值
│    · FastMoss 调用余量 / 本月成本
│    · 最近 10 条定价决策记录
│
├─ 2. 商品工作台（核心）⭐ MVP 主阵地
│    · 左：商品列表（我保存过的所有"候选/在售"商品）
│    · 右：详情页 = 成本四件套 + ROI 计算器 + 竞品分析 + 定价建议
│
├─ 3. 竞品探索
│    · 按关键词/类目查 FastMoss topSelling / product search
│    · 价格带分布图、销量分布、点进去看单品详情
│    · 一键"作为参考加入某商品的竞品池"
│
├─ 4. 运费与规则
│    · 运费表维护（普/特/敏货费率，支持后台更新）
│    · ROI 保本线、默认佣金率等全局参数配置
│
└─ 5. 设置
     · 店铺 OAuth 绑定、API key 管理、成员邀请
```

## 四、一阶段必须包含的 7 个功能（F1–F7）

### F1. 运费计算器 · 2 天
- 输入：产品重量 / 货物类型（普/特/敏，**运营手动选择**）/ 目的国（一阶段固定日本跨境直邮店铺）/ 是否包邮
- 输出：单件物流成本、不同打包组合（1 件/5 件/10 件装）的运费对比
- 运费表在管理后台维护（JSON 配置化，为未来扩展本土店/全托管店价卡预留）
- 价卡数据源：[`freight_jp_cross_border_direct_2025-08-27.json`](./freight_jp_cross_border_direct_2025-08-27.json)（已从 TikTok Seller University 抓取并结构化）

### F2. ROI 计算器 & 盈亏模拟 · 3 天
- 输入四项成本：货源成本（F6 图搜）/ 物流（来自 F1）/ 投流预算 / 达人佣金率（F7）
- **双向计算**：
  - 正向：给定售价 → 算 ROI
  - 反向：**给定目标 ROI（默认 2.5）→ 倒算广告费 + 达人分佣预算上限**（一阶段核心红线）
- 输出：零售价 / 折扣价 / 秒杀价 三档建议，盈亏平衡点
- 多 SKU 支持：单装 / 5 装 / 10 装分别算一组
- 退货率默认 4%、汇率默认 RMB:JPY=1:21，可配置

### F3. 竞品价格带分析 + 类目爆款榜 · 3 天
- 调用 FastMoss（截至 2026-04-09 已全部实测通过）：
  - `/product/v1/search`（关键词搜索，单 token）
  - `/product/v1/rank/topSelling`（日本爆款榜，含 **`growth_rate` 环比增长率作为选品核心维度**）
  - `/product/v1/goodSalesList`（28 天销量趋势 + 日销 ASP）
  - `/product/v1/productCategory`（类目树）
- 展示：价格分布直方图、Top 20 对标品、销量/价格散点图、爆款榜（按 growth_rate 排序）
- "加入参考池" → 在 F4 生成建议时作为上下文
- 已知 7 条硬限制见合同 §6.1 F3

### F4. LLM 智能定价建议 · 3 天
- 基于 F2 的成本结构 + F3 的竞品数据 + F7 的达人分佣预估 + 产品描述，调用 Claude Sonnet 4.6
- 输出**结构化建议**：
  - 建议零售价 / 折扣价 / 秒杀价（带理由）
  - 市场定位（低价引流 / 中位跟随 / 高价差异化）
  - 风险提示（如：竞品均价远低于保本价 → 建议弃坑）
- 同一产品可多次生成，保留历史建议对比
- 直连 Claude Sonnet 4.6 API（日本区无需 Proxy），开启 Prompt Caching 控成本
- 产品描述由运营手动粘贴（一阶段不做 TikTok 商品描述自动抓取）

### F5. 商品工作台 + 决策存档 + 运营回填实测 · 3.5 天 ⭐ 一阶段灵魂
- 创建 / 编辑 / 归档商品卡
- 每个商品保存：基础信息 · 成本结构 · F6 货源快照 · F3 竞品池快照 · F7 达人预估快照 · F4 历史定价建议 · 最终上架价
- **新增"运营回填"子功能**：商品上架后支持手动回填「最终上架价、上架后 N 天实测 GMV、实测 ROI、实际广告花费、实际达人分佣」，系统自动算出"AI 建议 vs 实际偏差"
- **这是数据资产沉淀的核心**：同一个品下次调整就不用重来
- 所有外部数据快照必须在查询当时入库（事后补齐不在一阶段范围）

### F6. 1688 图搜反查货源 · 1 天
- 万邦 Onebound `item_search_img`：上传一张 TikTok 竞品图 → 返回 1688 同款货源价
- `item_get` 拉 Top N 候选详情（price / priceRange / skus / seller_info / min_num）
- 作为 F2 货源成本的"外部参考值"
- **`item_weight` 为空时提示运营手填重量**
- **类目交叉验证**：1688 `cid` 对应类目名 vs FastMoss `category`，不匹配标记低置信度
- 价格阶梯使用"实际起订量对应阶梯"
- 图片经 AWS S3 + CloudFront 中转（Onebound 只接受公网 URL）
- Pangolin 备源只搭抽象层，不接实现

### F7. 达人分佣预估 · 3 天 + 1 天 Phase 0 预测试
- 调用 FastMoss：
  - `/product/v1/creatorList`（按 product_id 反查带货达人）
  - `/creator/v1/search`（达人画像，region=JP + category_id + follower_range）
  - `/creator/v1/productList`（按 uid 拉达人历史带货 SKU 及佣金率）
  - `/product/v1/rank/mostPromoted`（类目佣金率水位）
- 交付能力：
  - 选品后一键反查"带过同类竞品的日本达人 Top 20"
  - 每个达人展示粉丝数、28 天 GMV、互动率、历史平均佣金率
  - 运营指定"计划投 N 个达人" → 系统按类目佣金率水位 × 预期 GMV 算总分佣
  - 自动回填 F2 的"达人佣金"字段
- **Phase 0 前置 quota 测试**：若日本区达人 < 1,000 则降级为"按类目列 Top 50"
- 明确标注"**参考红线，非真实谈判价**"（FastMoss 返回的是商家公开佣金率，不代表达人实际谈判结果）

## 五、一阶段明确不做的功能（完整 25 项清单见合同 §6.1.2）

| 功能 | 原因 | 归属 |
|---|---|---|
| 投流优化助手 | TikTok Ads API 审批 1-3 周 + 需甲方先跑 1-2 个月投流积累训练数据 | Phase 3 |
| 达人评估系统完整版 | 需达人画像库 ≥1 万 + 60-90 天稳定性跟踪 + 达人 CRM | Phase 2 |
| 店铺自测 Dashboard（原 F6） | Phase 1 单店直接看 TikTok 商家后台；F5 的运营回填承接对比需求 | Phase 2+ |
| 标题/描述/视频 AI 生成 | 非 ROI 决策闭环 | Phase 2 |
| 浏览器插件（自动填字段） | 依附 DOM 脆弱 | Phase 4 |
| 订单智能分拣 | 履约环节 | Phase 4 |
| 活动到期提醒 / 续期 | 运维性质 | Phase 4 |
| 官方活动推荐 | 规则复杂 | Phase 4 |
| 多店铺 / 多租户 | 一阶段单店 | Phase 2+ |
| 多语言 UI / SSO / 审批流 | 非 ROI 闭环 | Phase 2+ |

## 六、关键设计决策

1. **"决策存档 + 运营回填"是灵魂**：每次定价建议都要可追溯、可复盘，并且运营能回填实测数据形成"建议 vs 实际"对比——这是一阶段核心数据资产。
2. **日本市场 + 跨境直邮单店**：一阶段不做多国 / 多店铺 / 本土店 / 全托管店价卡。
3. **FastMoss 按用量精算**：按 1,200 链接/月真实口径，含 30% 容灾 buffer，月预算 ≈$100。
4. **LLM 只用 Sonnet 4.6 + 少量 Haiku 4.5**：开启 Prompt Caching 控成本；日本区直连无需 Proxy。
5. **AI 建议准不准 ≠ 验收标准**：合同 §6.1.2.2 明确不把"AI 建议准确率"作为一阶段验收指标，靠 Phase 2 历史数据积累后再评估。

## 八、产品验收方式（非技术验收）

把本方案拿给：

1. **@孙栩尧（主用户）** — F1–F7 是否覆盖"新品上架前最痛的那 2 小时"？
2. **@创世跨境** — 一阶段的 7 个功能是否覆盖他关心的 ROI 决策链路？（Dashboard 延后，单店直接看 TikTok 商家后台即可）
3. **@李远哲** — 32 天工期 + ¥10,080 非人力成本 = 合同总额 ¥138,080，商务层面是否可接受？

### 一阶段量化验收指标
- 定价决策耗时：**从目前 ~120 分钟/品 降到 ≤ 20 分钟/品**
- 决策存档命中率：**≥ 50% 的新品决策能复用已存档商品的参数**
- F3/F6/F7 数据流稳定性：**≥ 99%**
- **不含"AI 建议准确率"指标**（合同 §6.1.2.2 明确）

## 九、风险

| 风险 | 影响 | 缓解 |
|---|---|---|
| 运营看到一阶段觉得"就这？没达人评估没投流优化" | 用户期望管理 | 本方案提前 review，明确 Phase 2/3 节奏；合同 §6.1.2.1/.2 写入共识 |
| FastMoss 日本市场数据深度不够 | F3 / F7 失效 | 2026-04-07 至 04-09 已实测：`/product/v1/search`、`/goodSalesList`、`/rank/topSelling JP total=500`、`/creator/v1/*` 全部可用 |
| FastMoss 日本达人池覆盖度未知 | F7 降级 | Phase 0 用 quota 预测试，若 < 1,000 则降级为"列 Top 50，不做筛选"，合同已写兜底 |
| Onebound `item_weight` 字段经常为空 | F6 → F1 → F2 不闭环 | 为空时提示运营手填重量，合同已写兜底 |
| 图搜跨类目召回（如手部按摩仪返回颈椎按摩仪）| F6 货源不准 | 类目交叉验证层，不匹配标记低置信度 |
| LLM 建议不准，用户不信 | F4 沦为摆设 | 建议页必须展示**推理过程**（成本结构 + 竞品摘要 + 达人预估），不只是结果 |
| 需求蔓延 | 工期超支、范围失控 | 严格按合同 §6.1.2 的 25 项不含清单执行，新需求按 §7 变更分级处理 |
