**6.1.2.1 关于"为什么不含"的说明（双方共识）**

Phase 1 的边界判定原则是：**只做能用现有第三方 API 直接取到数据的功能；凡是需要长期历史数据训练、跨账户数据聚合、或依赖甲方自有投放数据的功能，留到 Phase 2/3**。

- **投流优化**：TikTok Business API 的 `report/integrated/get/` 接口原生支持 `cpm` / `cpc` / `spend` 等字段，**但有两个硬限制**：（1）只返回经 OAuth 授权的**甲方自身广告账户**数据，无法查看竞品或市场类目均值；（2）必须甲方先真实投流才有数据，Phase 1 上线前样本不存在。加上 OAuth 审批 1-3 周会打乱 32 天工期，所以 Phase 1 不接 TikTok Ads API 自动化。**替代方案**：F2 ROI 计算器使用"倒算"逻辑——给定目标 ROI，系统输出「广告 + 达人分佣总预算上限」作为红线，运营拿着这个上限去 TikTok Ads Manager 后台设广告出价。Phase 1 上线 1-2 个月后，甲方自店投流数据积累起来，Phase 2 再考虑接 TikTok Ads API 做自动化刷新。

- **达人评估系统（完整版）**：FastMoss 接口能拉达人粉丝数、28 天 GMV、历史佣金率，这部分 Phase 1 的 F8 已经做了。**但做出完整版达人评估**，还需要：**达人实际谈判分佣率**（私下谈的，接口里只有商家公开佣金率）、**达人稳定性**（需要跟踪这个达人连续 60-90 天的数据变化）、**达人与甲方的合作履历**（需要甲方过往的合作记录入库）、**跨达人横向对比评分模型**（需要 ≥1 万达人的画像库）。数据接口只占工作量的约 20%，其余 80% 是画像库构建、匹配算法、评分模型训练，Phase 1 阶段不具备条件。

- **其他不含项**（内容生成 / 浏览器插件 / 订单分拣 / 活动提醒 / 多店铺 / 多语言 / 审批流 等）：与 ROI 决策闭环弱相关，属于"让运营工作更舒服"的效率工具或架构扩展，可延后到 Phase 2/3/4。

**6.1.2.2 双方对"准确性"的共识**

双方明确：Phase 1 交付的「AI 定价建议」「竞品数据展示」「达人分佣预估」「历史销量趋势」等，**其建议是否准确、是否有运营指导价值，需要依赖系统上线后甲方持续使用、积累自店历史数据（真实成交价、实跑 ROI、达人合作履历、投流花费等）来验证和迭代**。Phase 1 验收标准是"**功能闭环跑通 + 数据流可用 + 决策记录可存档**"，**不包含"AI 建议准确率"这类需要长时间数据验证的指标**。

通俗表述：Phase 1 是先把决策工具搭出来，让运营每次选品上品都有数据依据可查、有红线可守、有决策结果可存档。AI 建议的准不准，靠每天的真实使用慢慢校准——上线后 2-3 个月，甲方积累出"AI 建议价 vs 实际成交价 vs 实跑 ROI"的对比样本（由 F5 的运营回填子功能沉淀），才有条件做 Phase 2 的达人评估和投流优化模块。**在没有这批历史数据的前提下硬做"AI 投流优化"，输出结果不会比运营拍脑袋更准**。

6.1.3 **用户体系**：Phase 1 提供基础账号登录与权限控制，支持运营（主用户）/ 老板（只读看板）/ 管理员 三种角色；Phase 1 阶段面向单一店铺。

## 附录 A：FastMoss OpenAPI 接入契约（基于 2026-04-07 实测）

> 本附录是 F3 模块的技术契约。乙方按本附录实现，调用方式与字段命名以 FastMoss 官方文档（https://developers.fastmoss.com/zh/docs/）为准；如官方文档与本附录冲突，以官方文档为准并书面通知甲方。

**A.1 服务端基础信息**

| 项 | 值 |
|---|---|
| Base URL | `https://openapi.fastmoss.com` |
| Method | **POST**（非 GET） |
| Content-Type | `application/json` |
| 认证头 | `Authorization: Bearer <api_key>` |
| Client ID | 由甲方提供（生产环境正式 key 入 Secret Manager） |
| 计费方式 | 按接口按次，价格由 FastMoss 官方 SKU 决定 |

**A.2 Phase 1 接入接口清单**

| 接口 | 用途 | 实测状态 | 关键参数 |
|---|---|---|---|
| `/product/v1/search` | 关键词搜索商品（含价格、销量、评分、类目、店铺） | ✅ 已通 | `keywords`(string, 单 token) + `filter:{region}` + `orderby` + `page` + `pagesize` |
| `/product/v1/goodSalesList` | 商品 28 天销量趋势 + GMV + 关联视频 / 达人 | ✅ 已通 | `filter:{product_id, date_type}`，`date_type` 整数 1–28 |
| `/product/v1/rank/topSelling` | 日本区类目爆款榜 + `growth_rate` 环比增长率 | ✅ 已通 | `filter:{region, category_id, date_info}`，`date_info.type` ∈ `day/week/month`，`date_info.value` 具体日期字符串 |
| `/product/v1/creatorList` | **F8**：按 product_id 反查带货达人 | ✅ 已通 | `filter:{product_id}` + `orderby:{field: units_sold/gmv/follower_count/digg_count}` |
| `/creator/v1/search` | **F8**：达人画像搜索 | ⏳ Phase 0 预测试 | `filter:{region=JP, category_id, follower_range, verify_type, account_type}` |
| `/creator/v1/productList` | **F8**：达人历史带货 SKU 及佣金率 | ⏳ Phase 0 预测试 | `filter:{uid, days ∈ [0,7,14,28,90]}` |
| `/product/v1/rank/mostPromoted` | **F8**：类目佣金率水位参考 | ⏳ Phase 0 预测试 | `filter:{region, category_id, date_info}` + `orderby:{field: affiliate_count}` |
| `/product/v1/productCategory` | 类目树 | ⏳ 联调中 | 类目映射用 |
| `/product/v1/review` | 商品评论（可选） | – | F3 评分维度兜底 |

**A.3 已知限制（乙方在前端 / 服务端封装吸收）**

1. **`pagesize` 上限 10**：超过 10 返回空数组。前端分页或滚动加载封装由乙方处理。
2. **`keywords` 必须是单 token**：空格分隔的多关键词返回 `total=0`。前端搜索框需提示运营"用一个核心词"，或在服务端做关键词拆分多次调用合并去重。
3. **不存在 `/product/v1/detail` 按 ID 直取详情接口**：拿到 product_id 后想取详情，必须用 `/product/v1/search` 搜索命中，或调 `/product/v1/goodSalesList` 取 overview 字段。F5 商品工作台的"刷新最新价格"功能须基于此设计。
4. **`/product/v1/goodSalesList` 参数必须包在 `filter` 对象内**：直接传 `product_id`/`date_type` 在顶层会报 `date_type的值不能大于28`（误导性错误）。
5. **试用 quota 极小**：开发期估算 ~30 次调用就可能触发 `code=3 quota limit`。乙方须在开发环境用真实付费 key，并在服务端做 quota 监控告警。
6. **价格字段是字符串且可能含区间**：如 `"995円  -  1,442円"`，乙方须实现解析为 `[min, max]` 数值对。
7. **货币不固定**：日本站为 `JPY`，但同一接口返回多区域数据时 `currency` 字段不一定填充，乙方按 `region` 字段兜底判断。
8. **`/product/v1/rank/topSelling` 不返回 `price` 字段**：需按需反查 `/product/v1/search` 获取价格。`date_info` 只支持单日 / 单周 / 单月粒度，不做"近 N 天滑动窗口"聚合。
9. **`/product/v1/rank/topSelling` 文档未显式列出 JP 区**：但 2026-04-08 实测 `region=JP` 返回 total=500 有效数据，文档过时。
10. **`commission_rate` 字段格式不统一**：`/product/v1/search` 返回字符串（如 `"15%"`），`/creator/v1/productList` 与 `/product/v1/rank/mostPromoted` 返回整数（如 `1500` 表示 15%），乙方在解析层统一为百分数。
11. **"公开佣金率" ≠ "达人实际分佣率"**：API 返回的是商家设置的公开佣金率，不代表达人实际谈判结果。F8 的分佣成本估算值定位为"参考红线"，不是"真实谈判价"。

**A.4 容灾与监控要求**

- 所有 FastMoss 调用须经服务端中转（不在前端直接暴露 api_key）；
- 失败重试：网络错误重试 2 次（指数退避），业务错误（`code≠0`）不重试；
- quota 触发时（`code=3`），UI 显示降级提示并切换到本地缓存；
- 调用日志保留 ≥ 30 天，含请求体、响应 code、响应耗时；
- 缓存策略：搜索结果按 `(keywords, region, orderby, page)` 缓存 24 小时，销量趋势按 `(product_id, date_type)` 缓存 6 小时。

**A.5 验收时乙方提交的"接口实测报告"**

乙方在 F3 验收时须提交 `docs/data/fastmoss_smoke_test.md`，内容包括：
- 每个接入接口的至少 3 次真实调用记录（请求 + 响应截取）
- 失败/限流场景的实际响应样本
- 价格字段解析单元测试通过截图
